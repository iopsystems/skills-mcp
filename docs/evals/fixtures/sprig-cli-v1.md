## Repository Facts

- `sprig` is a small Rust CLI repository with `README.md`, a `clap` parser in
  `src/main.rs`, configuration handling in `src/config.rs`, nested
  parse/plan/apply modules rooted at `src/engine/mod.rs`, and CLI tests in
  `tests/cli.rs`.
- The canonical invocation is
  `sprig sync [--mode check|apply] [--format text|json] <CONFIG>`.
- `<CONFIG>` is a YAML file path. The file is a top-level mapping with required
  `source: <path>` and `destination: <path>` strings and an optional
  `exclude: [<glob>, ...]` list that defaults to empty.
- Defaults are `mode=check` and `format=text`.
- Missing config means an omitted or unreadable config and uses exit 2. Invalid
  schema means a YAML file that violates the schema and uses exit 3. Apply
  executes planned writes sequentially, may partially write, and then uses exit
  4 on a write failure.
- `--help` is generated by `clap`; actual rendered help is authoritative for
  user-visible syntax. The frozen rendered output is:

  ```text
  Usage: sprig sync [OPTIONS] <CONFIG>

  Arguments:
    <CONFIG>  Path to a YAML config file

  Options:
        --mode <MODE>      Operation mode [default: check] [possible values: check, apply]
        --format <FORMAT>  Output format [default: text] [possible values: text, json]
    -h, --help             Print help
  ```

- `src/main.rs` owns parsing and exit mapping; `src/config.rs` owns YAML loading
  and schema validation; `src/engine/parse`, `plan`, and `apply` own input
  discovery, side-effect-free planning, and sequential writes respectively.
  Check mode renders the plan without calling apply. `tests/cli.rs` locks the
  rendered help, defaults, modes, formats, and exits 2/3/4.
- The project stores the effort at
  `docs/journal/2026-07-14-atomic-apply.md`. Frontmatter is:

  ```yaml
  ---
  status: open
  opened: 2026-07-14
  updated: 2026-07-14
  ---
  ```

- Fixed lifecycle states are `status: open`, `status: shipped`, `status: no-go`,
  and `status: superseded`. A successfully verified `atomic-apply` effort moves
  from open to shipped in the same entry; it never uses a generic closed state.
- `docs/journal/index.md` uses the table columns `Opened | Effort | Status` and
  one row linking `2026-07-14-atomic-apply.md`; the row changes from open to
  shipped with the entry.
- Closure evidence uses `cargo test --locked`, `cargo run -- sync --help`, and
  focused invocations that demonstrate defaults and exits 2/3/4.
- The nested architecture or workflow may need a diagram. Three prior readers
